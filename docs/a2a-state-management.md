## Overview

The A2A protocol uses a **stateless HTTP/JSON-RPC transport** with **stateful Task objects**. State is NOT automatically maintained by the transport layer - instead, it must be explicitly managed through:

1. **Task IDs** - Unique identifier for each task/conversation
2. **Context IDs** - Groups related tasks together
3. **Task Store** - Server-side persistence layer
4. **Message History** - Accumulated conversation within a Task

The key insight is: **The client is responsible for providing the `task_id` in subsequent messages to continue a conversation.**

## Core Concepts

### The Task Object

The `Task` is the central state container in A2A. From `a2a/types.py`:

```python
class Task(A2ABaseModel):
    """Represents a single, stateful operation or conversation between a client and an agent."""
    
    id: str                          # Unique task identifier (UUID)
    context_id: str                  # Groups related tasks/interactions
    status: TaskStatus               # Current state (submitted, working, completed, etc.)
    history: list[Message] | None    # Conversation history
    artifacts: list[Artifact] | None # Generated outputs
    metadata: dict[str, Any] | None  # Extension data
```

### Task States

```python
class TaskState(str, Enum):
    submitted = 'submitted'       # Task received
    working = 'working'           # Agent processing
    input_required = 'input-required'  # Waiting for user input (multi-turn)
    completed = 'completed'       # Successfully finished
    canceled = 'canceled'         # User canceled
    failed = 'failed'             # Error occurred
    rejected = 'rejected'         # Agent refused task
    auth_required = 'auth-required'    # Needs authentication
    unknown = 'unknown'           # Client-side placeholder
```

## Key Identifiers: task_id and context_id

### task_id

- **Unique identifier** for a single task/conversation
- **Generated by the server** if not provided in the first message
- **Must be provided by the client** in subsequent messages to continue the conversation
- Located in `Message.task_id`

### context_id

- **Groups multiple related tasks** together
- Useful for scenarios where multiple sub-tasks belong to the same conversation
- **Generated by the server** if not provided
- Located in `Message.context_id`

### ID Generation (from `a2a/server/id_generator.py`)

```python
class UUIDGenerator(IDGenerator):
    """UUID implementation of the IDGenerator interface."""
    
    def generate(self, context: IDGeneratorContext) -> str:
        """Generates a random UUID, ignoring the context."""
        return str(uuid.uuid4())
```

### Step 1: Request Handling (DefaultRequestHandler)

When a message arrives, the `DefaultRequestHandler` sets up execution:

```python
# From a2a/server/request_handlers/default_request_handler.py

async def _setup_message_execution(
    self,
    params: MessageSendParams,
    context: ServerCallContext | None = None,
) -> tuple[TaskManager, str, EventQueue, ResultAggregator, asyncio.Task]:
    
    # Create TaskManager with task_id from the incoming message
    task_manager = TaskManager(
        task_id=params.message.task_id,      # May be None for first message
        context_id=params.message.context_id, # May be None for first message
        task_store=self.task_store,
        initial_message=params.message,
        context=context,
    )
    
    # Try to retrieve existing task
    task: Task | None = await task_manager.get_task()
    
    if task:
        # CONTINUING existing task - update history
        task = task_manager.update_with_message(params.message, task)
    elif params.message.task_id:
        # Task ID provided but task not found - ERROR
        raise ServerError(error=TaskNotFoundError(...))
    
    # Build request context (generates task_id/context_id if needed)
    request_context = await self._request_context_builder.build(...)
```

**Key Points:**
- If `task_id` is provided and exists → **Continue existing task**
- If `task_id` is provided but not found → **Error**
- If `task_id` is NOT provided → **Create new task with generated ID**

### Step 2: TaskManager - The State Coordinator

From `a2a/server/tasks/task_manager.py`:

```python
class TaskManager:
    """Manages a task's lifecycle during execution of a request."""
    
    def __init__(
        self,
        task_id: str | None,
        context_id: str | None,
        task_store: TaskStore,
        initial_message: Message | None,
        context: ServerCallContext | None = None,
    ):
        self.task_id = task_id
        self.context_id = context_id
        self.task_store = task_store
        self._initial_message = initial_message
        self._current_task: Task | None = None
    
    async def get_task(self) -> Task | None:
        """Retrieves current task from memory or store."""
        if not self.task_id:
            return None
        
        if self._current_task:
            return self._current_task
        
        # Load from persistent store
        self._current_task = await self.task_store.get(
            self.task_id, self._call_context
        )
        return self._current_task
    
    async def save_task_event(
        self, event: Task | TaskStatusUpdateEvent | TaskArtifactUpdateEvent
    ) -> Task | None:
        """Processes events and saves updated task state."""
        
        # Ensure task exists
        task: Task = await self.ensure_task(event)
        
        if isinstance(event, TaskStatusUpdateEvent):
            # Move current status message to history
            if task.status.message:
                if not task.history:
                    task.history = [task.status.message]
                else:
                    task.history.append(task.status.message)
            # Update status
            task.status = event.status
        
        # Persist to store
        await self._save_task(task)
        return task
```

### Step 3: TaskStore - Persistence Layer

The `TaskStore` interface (from `a2a/server/tasks/task_store.py`):

```python
class TaskStore(ABC):
    """Agent Task Store interface."""
    
    @abstractmethod
    async def save(self, task: Task, context: ServerCallContext | None = None) -> None:
        """Saves or updates a task in the store."""
    
    @abstractmethod
    async def get(self, task_id: str, context: ServerCallContext | None = None) -> Task | None:
        """Retrieves a task from the store by ID."""
    
    @abstractmethod
    async def delete(self, task_id: str, context: ServerCallContext | None = None) -> None:
        """Deletes a task from the store by ID."""
```

### Step 5: TaskUpdater - Agent Publishing Updates

The `TaskUpdater` (from `a2a/server/tasks/task_updater.py`) provides a convenient API for agents:

```python
class TaskUpdater:
    """Helper for agents to publish task updates."""
    
    def __init__(self, event_queue: EventQueue, task_id: str, context_id: str):
        self.task_id = task_id
        self.context_id = context_id
        self.event_queue = event_queue
    
    async def update_status(
        self,
        state: TaskState,
        message: Message | None = None,
        final: bool = False,
    ) -> None:
        """Publishes a TaskStatusUpdateEvent."""
        await self.event_queue.enqueue_event(
            TaskStatusUpdateEvent(
                task_id=self.task_id,
                context_id=self.context_id,
                final=final,
                status=TaskStatus(state=state, message=message, ...),
            )
        )
    
    async def requires_input(self, message: Message | None = None) -> None:
        """Marks task as needing more input (multi-turn)."""
        await self.update_status(TaskState.input_required, message=message)
```

### ClientTaskManager

From `a2a/client/client_task_manager.py`:

```python
class ClientTaskManager:
    """Manages task lifecycle on the client side."""
    
    def __init__(self) -> None:
        self._current_task: Task | None = None
        self._task_id: str | None = None
        self._context_id: str | None = None
    
    async def save_task_event(
        self, event: Task | TaskStatusUpdateEvent | TaskArtifactUpdateEvent
    ) -> Task | None:
        """Processes events and updates local task state."""
        
        if isinstance(event, Task):
            await self._save_task(event)
            return event
        
        # For update events, ensure we have a task object
        task = self._current_task
        if not task:
            task = Task(
                status=TaskStatus(state=TaskState.unknown),
                id=event.task_id,
                context_id=event.context_id,
            )
        
        if isinstance(event, TaskStatusUpdateEvent):
            # Add status message to history
            if event.status.message:
                if not task.history:
                    task.history = [event.status.message]
                else:
                    task.history.append(event.status.message)
            task.status = event.status
        
        self._current_task = task
        return task
```

     │  7. Response: Task with id & context_id      │

## State Storage Options

### InMemoryTaskStore (Default)

From `a2a/server/tasks/inmemory_task_store.py`:

```python
class InMemoryTaskStore(TaskStore):
    """In-memory implementation - data lost on restart!"""
    
    def __init__(self) -> None:
        self.tasks: dict[str, Task] = {}  # Simple dict
        self.lock = asyncio.Lock()
    
    async def save(self, task: Task, context=None) -> None:
        async with self.lock:
            self.tasks[task.id] = task
    
    async def get(self, task_id: str, context=None) -> Task | None:
        async with self.lock:
            return self.tasks.get(task_id)
```

**Pros:** Simple, fast
**Cons:** Data lost on restart, not suitable for multi-instance deployments

### DatabaseTaskStore (Persistent)

From `a2a/server/tasks/database_task_store.py`:

```python
class DatabaseTaskStore(TaskStore):
    """SQLAlchemy-based - supports PostgreSQL, MySQL, SQLite."""
    
    def __init__(
        self,
        engine: AsyncEngine,
        create_table: bool = True,
        table_name: str = 'tasks',
    ) -> None:
        self.engine = engine
        # ... SQLAlchemy setup
    
    async def save(self, task: Task, context=None) -> None:
        db_task = self._to_orm(task)
        async with self.async_session_maker.begin() as session:
            await session.merge(db_task)  # Upsert
    
    async def get(self, task_id: str, context=None) -> Task | None:
        async with self.async_session_maker() as session:
            stmt = select(self.task_model).where(self.task_model.id == task_id)
            result = await session.execute(stmt)
            task_model = result.scalar_one_or_none()
            return self._from_orm(task_model) if task_model else None
```

**Install with:** `pip install "a2a-sdk[sql]"` (or `postgresql`, `mysql`, `sqlite`)

## Architectural Decision: task_id vs Passing History in Payload

When implementing multi-turn conversations, you have two options:

### Option 1: Use `task_id` (A2A's Built-in State) ✅ RECOMMENDED

**How it works:**
- Client passes `task_id` in follow-up messages
- Server's `TaskStore` loads the `Task` with full `history`
- Agent accesses history via `context.current_task.history`

**Pros:**
- ✅ **Protocol-compliant** - Uses A2A as designed
- ✅ **Single source of truth** - State lives in one place (server)
- ✅ **Scalable** - Can swap `InMemoryTaskStore` → `DatabaseTaskStore` for persistence
- ✅ **Less payload size** - Only send new message, not entire history
- ✅ **Built-in features** - Task states, cancellation, resubscription work automatically
- ✅ **Multi-agent ready** - Other agents can query the same task via `tasks/get`

**Cons:**
- ❌ Client must track `task_id` per conversation
- ❌ `InMemoryTaskStore` loses state on restart (fixable with DatabaseTaskStore)

### Option 2: Pass Conversation History in Payload ❌ NOT RECOMMENDED

**How it works:**
- Client maintains full conversation history locally
- Each request includes the entire history in metadata/message
- Server is stateless, processes history each time

**Pros:**
- ✅ Server is truly stateless (easier horizontal scaling without shared storage)
- ✅ Client has full control over history

**Cons:**
- ❌ **Payload bloat** - History grows with each turn, increasing latency
- ❌ **Duplicated work** - Every agent must parse/validate history format
- ❌ **Not A2A-compliant** - Reinventing what the protocol already provides
- ❌ **Token waste** - If using LLMs, you're re-sending context that's already processed
- ❌ **No persistence** - If client disconnects, history is lost
- ❌ **Multi-agent complexity** - How do other agents access the conversation?

### Recommendation

**Use Option 1 (task_id)** because:

1. **You're using A2A** - Don't fight the framework
2. **History is already managed** - `DefaultRequestHandler` and `TaskManager` handle it automatically
3. **Future-proof** - When you add more agents, they can query task state via `tasks/get`
4. **Production-ready** - Swap to `DatabaseTaskStore` for persistence across restarts

### Streaming Messages: Extracting task_id

Yes, `task_id` is retrievable from streaming responses! Each streaming event contains `task_id`:

- `Task` → `task.id`
- `TaskStatusUpdateEvent` → `event.task_id`
- `TaskArtifactUpdateEvent` → `event.task_id`

**Complete Streaming Implementation (Using Modern ClientFactory API):**

> **Note:** `A2AClient` is deprecated. Use `ClientFactory.connect()` instead.

```python
import asyncio
from uuid import uuid4
from a2a.client import ClientFactory, ClientConfig
from a2a.types import (
    Message, TextPart, Part, Role, Task,
    TaskStatusUpdateEvent, TaskArtifactUpdateEvent, TaskState
)

class StreamingConversationClient:
    """Client that properly handles task_id across streaming multi-turn conversations."""
    
    def __init__(self, client):
        self.client = client
        self.task_id: str | None = None
        self.context_id: str | None = None
    
    async def send_message(self, text: str) -> str:
        """Send a message and return the agent's response, maintaining task_id for continuity."""
        
        # Build message - include task_id if we have one from previous turn
        message = Message(
            role=Role.user,
            parts=[Part(root=TextPart(text=text))],
            message_id=str(uuid4()),
            task_id=self.task_id,      # None on first message, set on follow-ups
            context_id=self.context_id,
        )
        
        agent_response_parts = []
        
        # Stream the response using the modern Client API
        async for event in self.client.send_message(message):
            # Handle tuple format: (Task, UpdateEvent | None)
            if isinstance(event, tuple):
                task, update = event
                
                # CRITICAL: Extract and save task_id from the FIRST event
                if self.task_id is None:
                    self.task_id = task.id
                    self.context_id = task.context_id
                    print(f"[Captured task_id: {self.task_id}]")
                
                # Process the update event
                if update is not None:
                    if isinstance(update, TaskStatusUpdateEvent):
                        # Status update - may contain a message
                        if update.status.message:
                            for part in update.status.message.parts:
                                if hasattr(part, 'root') and hasattr(part.root, 'text'):
                                    agent_response_parts.append(part.root.text)
                                elif hasattr(part, 'text'):
                                    agent_response_parts.append(part.text)
                    
                    elif isinstance(update, TaskArtifactUpdateEvent):
                        # Artifact update - process artifact parts
                        for part in update.artifact.parts:
                            if hasattr(part, 'root') and hasattr(part.root, 'text'):
                                agent_response_parts.append(part.root.text)
                
                # Check task state for conversation flow
                if task.status.state == TaskState.input_required:
                    print("[Agent waiting for more input]")
                elif task.status.state == TaskState.completed:
                    print("[Conversation completed]")
            
            # Handle direct Message response (non-streaming fallback)
            elif isinstance(event, Message):
                for part in event.parts:
                    if hasattr(part, 'root') and hasattr(part.root, 'text'):
                        agent_response_parts.append(part.root.text)
        
        return " ".join(agent_response_parts)
    
    def reset_conversation(self):
        """Start a new conversation by clearing task_id."""
        self.task_id = None
        self.context_id = None


# Usage Example with Modern ClientFactory API
async def multi_turn_streaming_example():
    """Example of a multi-turn streaming conversation using ClientFactory."""
    
    # Modern way: Use ClientFactory.connect() 
    # (A2AClient is deprecated)
    client = await ClientFactory.connect(
        agent="http://localhost:8001/",
        client_config=ClientConfig(streaming=True),
    )
    
    try:
        # Create conversation handler
        conversation = StreamingConversationClient(client)
        
        # Turn 1: Initial message (no task_id)
        print("User: I want to plan a trip to Tokyo")
        response1 = await conversation.send_message("I want to plan a trip to Tokyo")
        print(f"Agent: {response1}")
        print(f"[task_id after turn 1: {conversation.task_id}]")
        
        # Turn 2: Follow-up (task_id is now set automatically)
        print("\nUser: For 2 people, traveling in March")
        response2 = await conversation.send_message("For 2 people, traveling in March")
        print(f"Agent: {response2}")
        print(f"[task_id after turn 2: {conversation.task_id}]")  # Same task_id!
        
        # Turn 3: Another follow-up
        print("\nUser: Budget is around $5000")
        response3 = await conversation.send_message("Budget is around $5000")
        print(f"Agent: {response3}")
    
    finally:
        await client.close()


if __name__ == "__main__":
    asyncio.run(multi_turn_streaming_example())
```

**Key Points for Streaming:**

1. **First Event Contains task_id**: The very first streamed event (whether `Task` or update) contains the `task_id`
2. **Save It Immediately**: Capture `task_id` as soon as you receive the first event
3. **Reuse for Follow-ups**: Pass the saved `task_id` in subsequent messages
4. **Event Types in Streaming**:
   ```python
   # From a2a/types.py - what you receive in streaming:
   class TaskStatusUpdateEvent:
       task_id: str           # <-- task_id is here
       context_id: str
       status: TaskStatus
       final: bool
   
   class TaskArtifactUpdateEvent:
       task_id: str           # <-- task_id is here
       context_id: str
       artifact: Artifact
   ```

**Simplified Extraction Pattern:**

```python
async def extract_task_id_from_stream(client, message):
    """Extract task_id from the first streaming event."""
    async for event in client.send_message(message):
        if isinstance(event, tuple):
            task, _ = event
            return task.id  # task_id from first event
        elif isinstance(event, Message):
            return None  # Direct message response, no task created
    return None
```

## Common Pitfalls and Solutions

### Pitfall 1: Not Passing task_id in Follow-up Messages

**Symptom:** Each message creates a new task, conversation history is lost.

**Solution:** Extract `task_id` from the response and include it in subsequent messages:

```python
# After first message
response = await client.send_message(first_message)
task = response[0]  # Get the Task from response tuple
saved_task_id = task.id  # SAVE THIS!

# For follow-up
follow_up = Message(
    role=Role.user,
    parts=[...],
    message_id=str(uuid4()),
    task_id=saved_task_id,  # INCLUDE THIS!
)
```

### Pitfall 2: Using InMemoryTaskStore in Production

**Symptom:** Tasks disappear after server restart or can't be found across instances.

**Solution:** Use `DatabaseTaskStore`:

```python
from sqlalchemy.ext.asyncio import create_async_engine
from a2a.server.tasks import DatabaseTaskStore

engine = create_async_engine("postgresql+asyncpg://...")
task_store = DatabaseTaskStore(engine)
```

### Pitfall 3: Agent Not Accessing Conversation History

**Symptom:** Agent doesn't remember previous messages.

**Solution:** Use the `RequestContext.current_task` in your `AgentExecutor`:

```python
class MyAgentExecutor(AgentExecutor):
    async def execute(self, context: RequestContext, event_queue: EventQueue):
        # Get conversation history
        if context.current_task and context.current_task.history:
            for message in context.current_task.history:
                # Process previous messages
                text = get_message_text(message)
                print(f"Previous: {text}")
        
        # Current user input
        current_input = context.get_user_input()
```

### Pitfall 4: Not Handling input_required State

**Symptom:** Client doesn't know when to send follow-up messages.

**Solution:** Check the task state:

```python
async for event in client.send_message(message):
    if isinstance(event, tuple):
        task, update = event
        if task.status.state == TaskState.input_required:
            # Agent is waiting for more input
            # Prompt user and send follow-up with task_id
            pass
        elif task.status.state == TaskState.completed:
            # Conversation finished
            break
```

## Summary

| Component | Role in State Management |
|-----------|-------------------------|
| `task_id` | Unique identifier linking messages to a Task |
| `context_id` | Groups related tasks together |
| `Message.task_id` | **Client MUST set this** for follow-up messages |
| `Task.history` | Server maintains conversation history |
| `TaskStore` | Persists Task objects (in-memory or database) |
| `TaskManager` | Coordinates loading/saving Tasks during request |
| `TaskUpdater` | Agent uses this to publish state changes |
| `RequestContext.current_task` | Agent accesses history via this |

**The Golden Rule:** For multi-turn conversations, the client MUST:
1. Save the `task_id` from the first response
2. Include that `task_id` in all subsequent messages
3. The server will then load the existing Task with full history
