# Multi-Agent Travel Planner Application Design

## Overview
Build a multi-agent travel planner using Microsoft Agent Framework, MCP, and A2A protocol with:
- Centralized Orchestrator with **three-tier routing** (WORKFLOW → AGENT → TOOL) and shared state
- Pure parallel discovery phase execution
- Human-in-the-loop approval at 3 checkpoints (Clarification, Itinerary, Booking)
- Idempotent agents that work standalone with **Detect & Adapt** behavior
- Full lifecycle support: resume sessions, edit pending, modify confirmed bookings
- Pluggable storage interface (in-memory now, CosmosDB later)
- **End-to-end observability** with OpenTelemetry + Azure Monitor integration
- Design for Microsoft platform interoperability (M365 Copilot, Copilot Studio, Azure AI Agent Service)

---

## Key Design Decisions

### 1. Detect & Adapt Behavior
All agents auto-detect user input specificity:
- **Specific input** ("I want Hotel Gracery Shinjuku") → Validate, enrich with details, confirm
- **Vague input** ("find me a hotel") → Search and suggest options
- **Mixed input** ("a hotel near Shibuya, mid-range") → Search with constraints

**Implementation:** Each agent's LLM prompt includes specificity detection instructions:
```
If the user names a specific item (hotel name, flight number, attraction name):
  → Validate it exists, enrich with details (price, location, hours), return single result
If the user gives vague requirements or asks for suggestions:
  → Search for options matching criteria, return list of 3-5 options
If the user gives partial constraints:
  → Search with those constraints, return filtered list
```

**Orchestrator Hint (Optional):** Orchestrator can pass `{specificity: "specific", target: "Park Hyatt"}` to help agents skip detection.

### 2. Streamlined Approval Flow (3 Checkpoints, not 4)
- ~~Discovery approval~~ **REMOVED** - Discovery results flow directly into itinerary
- User sees combined results in context of the itinerary, not raw discovery data
- Faster flow: Clarify → Discover+Plan → Approve Itinerary → Book

### 3. Full Lifecycle Support
- **Resume sessions**: User gets session_id, can continue planning later
- **Edit pending bookings**: Modify cart items before final checkout
- **Modify confirmed bookings**: Change dates, cancel items after confirmation
- **Pluggable storage**: `SessionStore` interface with `InMemorySessionStore` now, `CosmosDBSessionStore` later

### 4. Single Booking Agent
Booking Agent handles full lifecycle: create, modify, cancel (not separate agents)

### 5. Agent Role Clarification (Avoiding Overlap)

| Agent | Responsibility | Does NOT Do |
|-------|---------------|-------------|
| **Aggregator** | Combines raw discovery outputs into unified structure. Calculates totals. | Does NOT validate against TripSpec |
| **Budget** | Proposes allocation, tracks spending, suggests reallocations | Does NOT validate non-financial constraints |
| **Validator** | Validates against TripSpec: dates covered, interests addressed, constraints respected. Calls Budget for cost check. | Does NOT calculate costs (delegates to Budget) |

**Flow:** Aggregator (combine + total) → Budget (allocate) → Route (plan) → Validator (check all constraints, including budget via Budget agent)

### 6. ID Model (Technical + Business Entities)

```
session_id (frontend conversation window)
    └── consultation_id (business: one active trip planning)
            ├── agent_tasks[] (A2A task_ids for each agent call)
            └── booking_ids[] (1:N bookings linked to this consultation)
```

| ID | Layer | Lifecycle | Purpose |
|----|-------|-----------|---------|
| **session_id** | Technical | Frontend-initiated | Conversation window, spans consultations over time |
| **consultation_id** | Business | Created at clarification start | Groups TripSpec, discovery, itinerary. Status: DRAFT → READY_TO_BOOK → PARTIALLY_BOOKED → FULLY_BOOKED |
| **booking_id** | Business | Created per booking | Individual booking (flight/hotel/event/pass). 1:N with consultation. Status: PENDING → CONFIRMED → MODIFIED → CANCELLED |
| **task_id** | Technical | Per A2A agent call | Internal A2A task, mapped to consultation_id |

**ID Generation Strategy:**
| ID | Generated By | Format | Example |
|----|--------------|--------|---------|
| session_id | Frontend | UUID v4 | `sess_a1b2c3d4-e5f6-...` |
| consultation_id | Orchestrator (on clarification start) | Prefixed UUID | `cons_x7y8z9...` |
| booking_id | Booking Agent | Prefixed UUID | `book_m1n2o3...` |
| task_id | A2A SDK | A2A standard | (internal) |

**Constraints:**
- One active consultation per session at a time
- User must complete/abandon consultation before starting new one
- Bookings can be added incrementally (book flight now, hotel later)

### 7. Consultation State Machine

```
                    ┌─────────────┐
                    │    DRAFT    │ ← Created when clarification starts
                    └──────┬──────┘
                           │ TripSpec approved
                           v
                    ┌─────────────┐
         ┌─────────>│  PLANNING   │ ← Discovery + Planning in progress
         │          └──────┬──────┘
         │                 │ Itinerary approved
         │                 v
         │          ┌─────────────┐
         │  modify  │READY_TO_BOOK│ ← User can start booking
         │  ┌──────>└──────┬──────┘
         │  │              │ First booking confirmed
         │  │              v
         │  │       ┌──────────────────┐
         │  └───────│ PARTIALLY_BOOKED │ ← Some bookings confirmed
         │          └──────┬───────────┘
         │                 │ All items booked
         │                 v
         │          ┌─────────────┐
         └──────────│ FULLY_BOOKED│ ← Complete (can still modify/cancel)
                    └──────┬──────┘
                           │
         ┌─────────────────┼─────────────────┐
         v                 v                 v
  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
  │  CANCELLED  │   │   EXPIRED   │   │  ARCHIVED   │
  │ (user abort)│   │ (TTL hit)   │   │ (completed) │
  └─────────────┘   └─────────────┘   └─────────────┘
```

**Transition Rules:**
- `FULLY_BOOKED` → `PARTIALLY_BOOKED`: When a booking is cancelled
- Any state → `CANCELLED`: User explicitly abandons
- Any state → `EXPIRED`: Session TTL exceeded (configurable, default 7 days)
- `FULLY_BOOKED` → `ARCHIVED`: After trip end date + 30 days

### 8. Three-Tier Routing Architecture

The Orchestrator uses a **three-tier routing system** to handle the full spectrum of user requests:

```
                    ┌─────────────────────────────────────────────┐
                    │           Unified Session State             │
                    │  (SessionManager + Consultation model)      │
                    │  All paths share the same state context     │
                    └─────────────────────────────────────────────┘
                                         │
                    ┌─────────────────────────────────────────────┐
                    │       Enhanced Intent Classifier            │
                    │  Returns: RoutingDecision + confidence      │
                    │  If confidence < 0.8: Ask user to clarify   │
                    └─────────────────────────────────────────────┘
                           │              │              │
              ┌────────────┘              │              └────────────┐
              ▼                           ▼                           ▼
    ┌──────────────────┐      ┌──────────────────┐      ┌──────────────────┐
    │   WORKFLOW PATH  │      │    AGENT PATH    │      │    TOOL PATH     │
    │   (Programmatic) │      │   (LLM-based)    │      │    (Direct)      │
    │                  │      │                  │      │                  │
    │ OrchestratorWork │      │ Route to agents  │      │ Direct tool call │
    │ flow phases:     │      │ for ad-hoc       │      │ No LLM reasoning │
    │ - CLARIFICATION  │      │ queries:         │      │                  │
    │ - DISCOVERY      │      │                  │      │ Tools:           │
    │ - AGGREGATION    │      │ "What's the best │      │ - currency_conv  │
    │ - ROUTING        │      │  time to visit?" │      │ - weather_lookup │
    │ - VALIDATION     │      │ "Tell me about   │      │ - timezone_info  │
    │ - BOOKING        │      │  visa rules"     │      │                  │
    │                  │      │ "Compare hotels" │      │                  │
    │ Triggers:        │      │                  │      │ Triggers:        │
    │ "Plan a trip"    │      │ Triggers:        │      │ "Convert 100 USD │
    │ "Book that"      │      │ Questions that   │      │  to JPY"         │
    │ "Continue"       │      │ don't fit flow   │      │ "Weather in      │
    └──────────────────┘      └──────────────────┘      │  Tokyo March"    │
              │                           │              └──────────────────┘
              │                           │                           │
              └───────────────────────────┴───────────────────────────┘
                                         │
                    ┌─────────────────────────────────────────────┐
                    │       State Update & Response Handler       │
                    │   Updates shared session state              │
                    │   Enables seamless path transitions         │
                    └─────────────────────────────────────────────┘
```

**Routing Decision Logic:**

| Path | When to Use | Examples |
|------|-------------|----------|
| **WORKFLOW** | Trip planning lifecycle, bookings, modifications to itinerary | "Plan a trip to Tokyo", "Book the flight", "Change the hotel" |
| **AGENT** | Questions, advice, comparisons outside structured workflow | "What's the weather like in March?", "Is Tokyo expensive?", "Compare these two hotels" |
| **TOOL** | Simple lookups that don't need LLM reasoning | "Convert 500 USD to EUR", "What time is it in Tokyo?", "Weather forecast Nov 10" |

**Ambiguity Handling:**
- When intent confidence < 0.8, ask user: "Are you planning a trip or just asking a question?"
- Example: "Tell me about hotels in Tokyo" → Could be WORKFLOW (start planning) or AGENT (just information)

**State Sharing:**
- All three paths read from and write to the same `SessionManager` state
- Agent/Tool responses are stored in `conversation_context` within the session
- Workflow can reference context from earlier Agent queries (e.g., user asked about Tokyo weather → later plans trip there)

**Path Transitions (Auto-Detected):**
- User asks question (AGENT) → "Book that for me" → Transitions to WORKFLOW
- User in WORKFLOW → "Actually, what's visa policy?" → Temporarily AGENT → Resume WORKFLOW
- TOOL response triggers planning: "Weather is bad in March" → "Plan for April instead" → WORKFLOW

**Implementation Components:**
- `src/agents/orchestrator_agent/routing_dispatcher.py` - Routes to appropriate path
- `src/agents/orchestrator_agent/tool_executor.py` - Direct tool invocation (currency, weather, timezone)
- Enhanced `IntentClassifier` with `RoutingDecision` enum

## Error Handling Strategy

### Parallel Discovery Failures (5 agents: POI, Stay, Transport, Events, Dining)
| Scenario | Strategy |
|----------|----------|
| 1 of 5 agents fails | **Proceed with partial results**. Mark missing category. Inform user: "Couldn't find events, but here's everything else." |
| 2+ agents fail | **Retry once**, then ask user: "Having trouble searching. Try again or proceed with limited results?" |
| All agents fail | **Halt and inform user**. Offer retry or manual input. |
| Agent timeout (30s) | Treat as failure. Use circuit breaker for repeated timeouts. |

### Validation Failures
| Issue Type | Strategy |
|------------|----------|
| Over budget | Validator returns `{passed: false, issues: ["Over budget by $500"]}`. Orchestrator auto-triggers Budget agent with `mode: REALLOCATE` to suggest alternatives. |
| Missing dates | Validator flags. Route agent re-runs with explicit gap-filling. |
| Constraints violated | Return to user with explanation: "Couldn't find vegetarian options for Day 3. Approve anyway or modify?" |

### Booking Failures
| Scenario | Strategy |
|----------|----------|
| Single booking fails | Mark as FAILED, continue with others. User can retry or skip. |
| Payment fails | Halt checkout, preserve cart. Do NOT mark as confirmed. |
| External API down | Circuit breaker after 3 failures. Inform user, offer to save for later. |
| Partial checkout (2/3 booked, 1 fails) | Mark consultation as `PARTIALLY_BOOKED`. User can retry failed item. |

### Intent Classification Uncertainty
| Confidence | Action |
|------------|--------|
| > 0.8 | Proceed with classified intent |
| 0.5 - 0.8 | Confirm with user: "Did you mean to search for hotels?" |
| < 0.5 | Ask clarifying question: "I'm not sure what you'd like to do. Could you rephrase?" |

### Compensation/Rollback
- **No automatic rollback** for confirmed bookings (external systems)
- User must explicitly cancel each booking
- System tracks which bookings are confirmed vs pending
- For demo (mock provider): Can simulate rollback

## Caching Strategy (for Targeted Re-run)

### What Gets Cached
| Data | Cache Location | TTL | Invalidation Trigger |
|------|---------------|-----|---------------------|
| Discovery results (POI, Stay, Transport, Events, Dining) | Consultation object in ConsultationStore | Until consultation expires | TripSpec change (dates, destination, interests, dietary) |
| Aggregated results | Consultation.discovery_results | Same as above | Any discovery result changes |
| Budget allocation | Consultation.budget_tracking | Same as above | Price changes, item additions |

### Cache Key
```
cache_key = f"{consultation_id}:{agent_name}:{hash(relevant_trip_spec_fields)}"
```

### Invalidation Rules
| Change | Invalidates |
|--------|-------------|
| Destination change | ALL discovery caches |
| Date change | ALL discovery caches (availability changes) |
| Budget change | Budget + Validator only |
| Interest change | POI + Events caches |
| Dietary constraint change | Dining cache only |
| "Change hotel" request | Stay cache only |
| "Change restaurant" request | Dining cache only |

### Implementation
- **Phase 1 (Demo):** No caching. Always re-run full pipeline. Simple but slower.
- **Phase 2 (Optimization):** Cache in Consultation object. Targeted re-run uses cached results for unchanged agents.

## Sample Workflow with Human-in-the-Loop (3 Checkpoints)

### Phase 1: Clarification
```
USER: "I want to plan a trip to Tokyo"
[Orchestrator] -> Intent: START_TRIP_PLANNING -> Route to Clarifier
[Clarifier] <- -> Multi-turn conversation to gather TripSpec

>>> HUMAN APPROVAL #1: Confirm TripSpec <<<
[Orchestrator] "I've gathered: Tokyo, Nov 10-17, 2 people, $3000/person..."
[Approve] [Modify] [Cancel]
```

### Phase 2: Discovery + Planning + Validation
```
[Orchestrator] Spawns 5 parallel A2A calls:
  - POI Agent (10008)
  - Stay Agent (10009)
  - Transport Agent (10010)
  - Events Agent (10011)
  - Dining Agent (10017) <- Finds restaurants matching dietary constraints
[Aggregator] Combines all results
[Budget Agent] Creates budget allocation
[Route Agent] Builds day-by-day itinerary
[Validator Agent] Self-checks against TripSpec:
  ✓ Total $4,454 within $6,000 budget
  ✓ All 7 days covered
  ✓ Art, food, temples interests addressed
  ✓ Vegetarian constraint noted in restaurant picks

>>> HUMAN APPROVAL #2: Validated Itinerary <<<
[Orchestrator] "Here's your 7-day Tokyo itinerary (validated):
  Day 1: Arrive NRT (ANA $1,247/pp), check into Hotel Gracery ($240/night)
  Day 2: AM: Senso-ji Temple, PM: Ueno Museums, EVE: Akihabara
  ...
  Total: $4,454 (Budget: $6,000 - within budget)"
[Approve & Book] [Modify Day X] [Change Hotel] [Start Over]

# If user says "Change Hotel" -> Targeted Re-run:
[Orchestrator] Detects modification target: Stay
[Stay Agent] Re-runs with new constraint/preference
[Aggregator] Re-combines (uses cached POI/Transport/Events)
[Budget Agent] Recalculates with new hotel price
[Route Agent] Updates itinerary with new hotel
[Validator Agent] Re-validates
>>> Back to HUMAN APPROVAL #2 <<<
```

### Phase 3: Booking
```
>>> HUMAN APPROVAL #3: Each Booking Item <<<
[Orchestrator] "Ready to book:
  1. Flight: ANA NH106 MEL->NRT $1,247/pp [Book] [Skip] [Alternative]
  2. Hotel: Gracery Shinjuku 7 nights $1,680 [Book] [Skip] [Alternative]
  3. JR Pass: 7-day $280/pp [Book] [Skip]"

[User selects items to book]

>>> FINAL CHECKOUT <<<
[Orchestrator] "Confirm bookings: Flight + Hotel + Pass = $4,454"
[Confirm All] [Cancel All] [Review Individual]
```

### Lifecycle: Resume, Edit, Modify
```
# Resume a session
USER: "Continue my Tokyo trip planning" (provides session_id)
[Orchestrator] Loads session state, resumes from last checkpoint

# Edit pending booking
USER: "Change the hotel to Park Hyatt"
[Orchestrator] -> Stay Agent (Detect & Adapt: specific hotel request)
[Stay Agent] Validates Park Hyatt, returns enriched details
[Orchestrator] Updates pending bookings, recalculates budget

# Modify confirmed booking
USER: "I need to cancel the JR Pass booking"
[Orchestrator] -> Booking Agent (mode: CANCEL)
[Booking Agent] Processes cancellation, returns confirmation
```

## Intent Classification (Orchestrator)

### Routing Decision (Three-Tier)

```python
class RoutingDecision(str, Enum):
    """Top-level routing decision for incoming messages."""
    WORKFLOW = "workflow"   # Route to OrchestratorWorkflow (programmatic)
    AGENT = "agent"         # Route to specific agent (LLM-based)
    TOOL = "tool"           # Direct tool invocation (no LLM)
    CLARIFY = "clarify"     # Confidence too low, ask user

class RoutingResult(BaseModel):
    """Result of intent classification."""
    decision: RoutingDecision
    confidence: float                    # 0.0 - 1.0
    intent: IntentType | None            # Specific intent if determined
    target_agent: str | None             # Agent name for AGENT path
    target_tool: str | None              # Tool name for TOOL path
    clarification_prompt: str | None     # Question to ask if CLARIFY

# Routing rules
WORKFLOW_INTENTS = {
    IntentType.START_TRIP_PLANNING,
    IntentType.CONTINUE_CLARIFICATION,
    IntentType.APPROVE_TRIP_SPEC,
    IntentType.APPROVE_ITINERARY,
    IntentType.START_BOOKING,
    IntentType.CONFIRM_BOOKING,
    IntentType.RESUME_SESSION,
    IntentType.EDIT_PENDING,
    IntentType.MODIFY_BOOKING,
    IntentType.CANCEL_BOOKING,
    IntentType.MODIFY_TRIP,
}

AGENT_INTENTS = {
    IntentType.SEARCH_POI,
    IntentType.SEARCH_STAY,
    IntentType.SEARCH_TRANSPORT,
    IntentType.SEARCH_EVENTS,
    IntentType.SEARCH_DINING,
    IntentType.SPECIFY_ITEM,
    IntentType.CHECK_BUDGET,
    IntentType.GENERAL_QUESTION,  # New: general travel questions
}

TOOL_INTENTS = {
```


### PII Handling
| Data | Classification | Handling |

- **Demo:** No auth, all sessions accessible. Clearly mark as non-production.

```python
class A2AClientWrapper:
    async def send_message(self, agent_url: str, message: Message) -> TaskResponse:
        headers = {}
```

```python
import logging
import json
from opentelemetry import trace

class StructuredFormatter(logging.Formatter):
    def format(self, record):
        span = trace.get_current_span()
        ctx = span.get_span_context()

        log_record = {
            "timestamp": self.formatTime(record),
            "level": record.levelname,
            "message": record.getMessage(),
            "logger": record.name,
            "trace_id": format(ctx.trace_id, '032x') if ctx.is_valid else None,
            "span_id": format(ctx.span_id, '016x') if ctx.is_valid else None,
        }

        # Add custom attributes if present
        if hasattr(record, 'task_id'):
            log_record['task_id'] = record.task_id
        if hasattr(record, 'session_id'):
            log_record['session_id'] = record.session_id

        return json.dumps(log_record)
```

## Interoperability Design

### Scope
**Phase 1 (Current):** All agents are homogeneous (same framework, same protocol). Focus on A2A protocol compliance.

**Phase 2 (Future):** Swap individual agents with external platform agents.

### Agent Swappability Requirements
For an external agent to replace an internal one:

1. **Must implement A2A protocol** - Same `/messages/send`, `/tasks/get` endpoints
2. **Must return compatible Pydantic models** - Same response schema
3. **Must expose AgentCard** - For Orchestrator discovery

### Platform Adapter Pattern (Phase 2)
```python
class PlatformAdapter(ABC):
    """Translates between A2A and platform-specific protocols."""

    @abstractmethod
    async def translate_request(self, a2a_request: MessageSendParams) -> PlatformRequest: ...

    @abstractmethod
    async def translate_response(self, platform_response: PlatformResponse) -> A2AResponse: ...

# Example: Swap Stay Agent with Copilot Studio agent
agent_registry.register(
    name="stay",
    url="https://copilot-studio.example.com/stay-agent",
    adapter=CopilotStudioAdapter()
)
```

### What Adapters Need to Handle
| Platform | Translation Needed |
|----------|-------------------|
| M365 Copilot | Graph API auth, Teams message format |
| Copilot Studio | Power Platform connectors, Dataverse |
| Azure AI Agent Service | Managed identity, Azure-specific headers |

**Note:** Adapter implementation is out of scope for Phase 1. Define interfaces only.

## Booking Modification Constraints (Real-World)

### Mock Provider (Demo)
- All modifications allowed instantly
- No fees, no restrictions

### Real Provider Considerations
| Booking Type | Modify Constraints | Cancel Constraints |
|--------------|-------------------|-------------------|
| Flights | Often not modifiable (must cancel + rebook) | Refund depends on fare class |
| Hotels | Usually changeable until X days before | Cancellation policy varies |
| Events | Usually non-refundable | May allow name change |
| Transport passes | Non-modifiable | Refundable before activation |

### Design Implications
```python
class Booking(BaseModel):
    ...
    can_modify: bool = True          # Provider tells us
    can_cancel: bool = True
    cancellation_policy: str | None  # "Free until 48h before"
    modification_fee: float | None   # e.g., $150 change fee
```

### User Experience
- Show modification constraints at booking time: "This flight cannot be changed after booking"
- On modify request: Check `can_modify` first, inform user if not possible
- On cancel request: Show refund amount before confirming
